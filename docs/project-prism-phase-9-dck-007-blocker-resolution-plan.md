## Current Facts

The likely ownership mismatch is in root tab-drag session continuity:

- `FloatingWindowComponent` treats tab drag as session-owned. Once
  `#draggingTab` is true, `onInputMove` calls `moveTabDrag(event.point)` even
  when the pointer leaves the tab hit.
- `WorkspaceRootDockFrameComponent` still calls `moveTabDrag(event.point)` only
  when the current move hit is still `WINDOW_FRAME_TAB_PART_ID`.
- The failed browser path starts from a root `Scene` tab and moves into a
  `Debug` pane. That path can leave the tab hit before preview generation, so
  the tab drag session may never advance to a preview/commit.

This is a UI dock owner behavior problem, not an app-composition or product
installer ownership problem.

## Non-Negotiables

- Do not add a compatibility dock path.
- Do not add a test-only fake facade to make smoke pass.
- Do not make Debug Log, smoke evidence, or app composition a placement owner.
- Do not patch the graph reducer unless evidence shows preview/intent/commit
  reached lifecycle and graph validation failed.
- Do not reintroduce product-level feature policy, hierarchy metadata, or
  workspace-mode compatibility while fixing DCK-007.
- Prefer making root and floating frame tab-drag semantics identical over
  adding another branch.

## Step 2: Fix Root Tab Drag Session Continuity

Purpose: make root and floating frame tab-drag semantics match.

Work:

1. Change `WorkspaceRootDockFrameComponent.onInputMove` so an active tab drag
   session owns subsequent move routing:

```text
if (this.#draggingTab) {
  this.#tabDragSink?.moveTabDrag(event.point);
  return;
}
```

2. Preserve splitter behavior for actual splitter sessions. Do not convert root
   content drag into window movement or placement mutation.
3. Keep `handleWindowFrameTabInputEnd` as the single tab input commit owner.
4. Update `onInputCancel` to cancel when `#draggingTab` is true, not only when
   the cancel hit is a tab.
5. Add an owner-level regression equivalent to the existing floating-frame
   regression:
   - start on root `Scene` tab;
   - move with a non-tab hit into the `Debug` pane target;
   - release/end with a non-tab hit over the `Debug` pane or tabset target;
   - do not satisfy this test with a tab-internal move/end hit;
   - assert `moveTabDrag` is called;
   - assert a same-frame split/merge commit intent is submitted;
   - assert root frame did not use a fallback mutation path.

Exit:

- root tab drag preview continues after leaving the tab hit.
- root and floating frame components share the same session-owned tab drag
  semantics.

Validation:

